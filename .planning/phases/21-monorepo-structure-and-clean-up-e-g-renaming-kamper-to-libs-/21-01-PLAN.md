<objective>
Rename `kamper/` to `libs/` at the filesystem level, update `settings.gradle.kts` to use `:libs:*` project paths, and update every `project(":kamper:*")` reference in library and demo build scripts so the build is green again. This is the single build-breaking unit of work — all changes in this plan must land in one commit.

Purpose: The monorepo top-level directory `kamper/` shadows the published artifact namespace and is unconventional for a library monorepo (D-10 research validated). Renaming to `libs/` produces a cleaner, idiomatic top-level layout while leaving Maven coordinates `com.smellouk.kamper:*` and plugin IDs unchanged (D-06).

Output: A renamed directory tree plus 23 updated build files, all landing in one atomic commit, with `./gradlew test` green.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-21-01 | Tampering | settings.gradle.kts (build graph integrity) | mitigate | Atomic commit: filesystem rename and `include()` updates land together; verify with `./gradlew :libs:api:test` before commit. Pitfall 1 in 21-RESEARCH.md is the primary risk vector; smoke test catches it. |
| T-21-02 | Information Disclosure | published Maven artifact IDs | accept | D-06 / ADR-004 explicitly preserve `com.smellouk.kamper:*` coordinates; KamperPublishPlugin artifactId derives from `project.projectDir.parent` directory name (`modules`/`integrations`), which is rename-safe per 21-RESEARCH.md verification. First post-rename build is slower but functional. No mitigation needed — well-documented Gradle behavior. |
| T-21-04 | Repudiation | git rename history | mitigate | Use `git mv kamper libs` (not OS `mv`) so git records the rename event and history is traceable per file. |
